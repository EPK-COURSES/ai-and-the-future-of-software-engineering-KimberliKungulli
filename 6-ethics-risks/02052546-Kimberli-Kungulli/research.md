## Introduction

Artificial intelligence tools have become deeply embedded in modern software development. According to a recent developer survey of 49,000 respondents, 84% of developers now use AI coding assistants, with more than half relying on them daily (Kusari, 2026). Google's 2024 State of DevOps Report found that 74.9% of respondents already use AI in software development (ACM, 2025).

This rapid adoption brings real productivity gains, but also serious risks that are not always visible to developers in the moment of use. This research explores the most significant risks, ethical concerns, and technical limitations that software engineers must understand and actively manage.

---

## 1. Hallucinations and Incorrect Code Generation

One of the most documented and dangerous limitations of AI coding assistants is the phenomenon known as **hallucination** — where the model confidently generates output that is factually wrong or completely fabricated.

In software development, hallucinations take a particularly dangerous form. Research published at USENIX describes "package hallucinations," where an AI coding assistant recommends a third-party software package that simply does not exist (USENIX, 2025). Developers who follow this recommendation may unknowingly open the door to supply chain attacks.

A 2025 study found that nearly 20% of packages recommended by large language models (LLMs) did not actually exist (Kusari, 2026). Attackers have begun systematically registering these AI-hallucinated package names — a technique now called **slopsquatting** — creating traps for developers who trust the AI output without verification (Snyk, 2025).

Beyond package hallucinations, AI-generated code itself frequently contains vulnerabilities. A study by Fu et al. (2024) found that out of 452 real-world code snippets generated by GitHub Copilot, **32.8% of Python and 24.5% of JavaScript snippets** contained security weaknesses — including eight vulnerability types from the 2023 CWE Top 25 list (CSET, 2024). Separately, a previous study cited by Trend Micro found that **40% of code generated by GPT models contained vulnerabilities** (Trend Micro, 2024).

Professor Michael Pradel from the University of Stuttgart, a software engineering expert, noted that "models are trained to always say something — they are rarely trained to say 'I don't know.'" He observed that when a model starts hallucinating, it can take a developer more time to fix the generated code than it would have taken to write it manually (ACM, 2025).

## 2. Security Vulnerabilities in AI-Assisted Development

AI-generated code introduces a new class of security risks that traditional code review was not designed to catch.

According to research aggregated by diffray.ai, **29–45% of AI-generated code contains security vulnerabilities**, and nearly 20% of package recommendations point to libraries that do not exist (diffray, 2026). A JetBrains 2024 survey found that **59% of developers lack trust in AI-generated code for security reasons**, and 42% reported ethical concerns about AI tool use — with 28% of companies actively limiting AI tool use in their workflows (diffray, 2026).

The CSET (Center for Security and Emerging Technology) November 2024 issue brief identified several specific threat categories in AI-generated code:
- Code that reproduces vulnerable patterns from training data (Common Weakness Enumerations, CWEs)
- References to nonexistent, outdated, or malicious external libraries
- Slopsquatting supply chain attacks through hallucinated package names (CSET, 2024)

The core problem is that AI tools learn from existing codebases, including codebases that contain bugs and vulnerabilities. They reproduce these patterns without understanding why the code is dangerous.

## 3. Overdependence on AI and Skill Degradation

A more subtle but long-term risk is the effect of AI dependency on developer skill. When developers routinely accept AI-generated code without deeply understanding it, their own ability to write, debug, and reason about code may weaken over time.

The Information Difference (2025) describes "vibe coding" — where novice developers use AI to generate entire programs without understanding the underlying logic. They note that "debugging code that you didn't write yourself is clearly harder than something that you wrote and understood the logic for" — meaning that even if code is generated faster, the process of testing and debugging may take longer (The Information Difference, 2025).

The ASCE (American Society of Civil Engineers), writing about AI ethics in engineering professions broadly, warned that overreliance on AI tools may entice engineers to "exceed their technical competence" and may lead to "careless or ill-informed decisions that threaten the health or safety of persons who depend on the engineer's services" (ASCE, 2025). While this was written for civil engineers, the principle applies directly to software engineers who ship systems used in healthcare, finance, transportation, and critical infrastructure.

The deeper question raised by the assignment brief — *What happens if engineers stop understanding the code they ship?* — is not hypothetical. As AI takes over more of the routine coding work, the risk of engineers who cannot independently verify, debug, or redesign that code becomes a professional and societal concern.

## 4. Ethical Responsibility and Accountability

When an AI tool generates code that causes a security breach or system failure, the question of accountability becomes complicated. Is it the developer who accepted the suggestion? The company that deployed the tool? The vendor that built the AI? Currently, there is no clear legal answer.

The EU AI Act, which entered into force in August 2024 and became effective in 2025, mandates strict documentation for AI use in critical sectors such as healthcare (Medium/CreedTec, 2025). This represents an early attempt to assign regulatory responsibility, but it covers only certain high-risk domains and certain jurisdictions.

People Tech Group (2025) identifies ethical challenges that are growing in importance as AI becomes more powerful: bias in decision-making systems, privacy misuse through mass data collection, and opaque decisions where neither the user nor the developer can explain why the AI produced a specific output. They argue that "ethics are no longer optional; they are critical to ensuring AI benefits society without harm."

Henry Ly's analysis in Predict/Medium notes that companies which establish structured ethical AI practices — including clear dataset selection guidelines, model explainability requirements, and fairness audits — are 2.5x more likely to build customer trust (Accenture, cited in Ly, 2024).

## 5. Copyright and Intellectual Property Issues

AI models for code generation are trained on vast datasets of publicly available code, much of it licensed under open-source terms. Whether this training constitutes copyright infringement, and whether AI-generated code inherits the licensing obligations of its training data, remains legally unresolved.

A 2024 survey of 574 GitHub developers (Stalnaker et al., 2024) found that developer opinions on copyright issues vary broadly. While many developers are aware of the nuances involved, significant uncertainty and misconceptions remain — particularly around the question of who owns code that an AI assistant generated.

The U.S. Copyright Office published Part 2 of its Copyright and Artificial Intelligence report in January 2025, addressing the copyrightability of AI-generated works. In March 2025, the U.S. Court of Appeals for the D.C. Circuit affirmed that AI systems cannot be listed as authors under copyright law (Congress.gov, 2025). This leaves important questions open: if the AI cannot be the author, is the developer the author? Does the code inherit the license of training data the AI was exposed to?

The Generative AI Copyright Disclosure Act of 2024, introduced in the U.S. Congress, would require companies to disclose the datasets used to train their AI systems — a step toward greater transparency (USC IPTLS, 2025). Until such requirements exist globally, software engineers working with AI tools operate in legal uncertainty about the intellectual property status of the code they produce.

## 6. Privacy Concerns

AI coding assistants often operate by sending code context — sometimes including proprietary business logic — to remote servers for processing. This creates a privacy risk, especially in enterprise or regulated environments.

People Tech Group (2025) notes that AI systems collect vast amounts of data to deliver personalized and accurate responses. When developers use AI coding tools in sensitive environments — working on financial systems, medical software, or security-critical infrastructure — there is a real risk that proprietary code or sensitive data is processed by third-party AI services that may retain or analyze that data.

This concern explains why 28% of companies surveyed by JetBrains in 2024 actively limit or restrict AI tool use in their development workflows — not because the tools are ineffective, but because of security and privacy concerns (diffray, 2026).

---

## 7. Technical Debt from AI-Generated Code

AI tools can generate syntactically correct code quickly, but speed does not guarantee long-term maintainability. Code that passes tests at the moment of generation may introduce hidden architectural problems — what the software engineering community calls **technical debt**.

The Information Difference (2025) notes that LLMs trained on code libraries from years ago may produce code using outdated libraries that have since developed security vulnerabilities. Gartner predicted that by 2026, 60% of AI projects would be abandoned due to poor-quality data — a signal that AI-generated outputs, including code, require sustained human oversight to remain viable (People Tech Group, 2025).

A 2024 trial cited by CreedTec showed that agentic AI systems misinterpreted ambiguous requirements in 15% of cases. In a large codebase, even a small percentage of misinterpreted requirements can accumulate into significant technical debt over time (Medium/CreedTec, 2025).

## Conclusion

The risks, ethics, and limitations of AI in software engineering are not theoretical concerns for the future — they are active challenges that the industry faces today. From hallucinated packages that enable supply chain attacks, to AI-generated code with known vulnerability types, to the gradual erosion of developer expertise through overdependence, the engineering profession faces risks that require deliberate and informed responses.

Responsible use of AI in software engineering requires: critical review of all AI-generated code before acceptance, awareness of the security and legal landscape, maintenance of strong fundamental skills independent of AI assistance, and participation in the emerging ethical frameworks and regulations that govern how these tools are built and deployed.

The most important takeaway from this research is that AI tools are not replacements for engineering judgment. They are powerful aids that require skilled human oversight to remain safe and effective.

## 8. Real-World Incidents Caused by AI-Generated Code

The risks discussed in this research are not theoretical. Real incidents have already occurred.

Aikido Security's *State of AI in Security & Development 2026* report, surveying 450 developers and security engineers across Europe and the US, found that **1 in 5 organizations suffered a major breach directly linked to AI-generated code**. The same report found that 45% of AI-generated code samples failed basic security tests, with Java showing a 72% failure rate (Aikido/RGCS, 2025).

A December 2025 study by CodeRabbit analyzed 470 real-world open-source pull requests and found that AI-generated code introduces **2.74x more security vulnerabilities** and 1.7x more total issues than human-written code (Docker, 2026).

In one widely discussed incident from December 2025, a developer asked an AI coding agent to clean up packages in a repository. The agent interpreted the instruction too broadly and deleted the user's entire home directory — including desktop files, documents, and authentication credentials. Recovery was impossible. The incident was covered by multiple outlets and highlighted what happens when AI tools are given broad permissions without human oversight (Docker, 2026).

According to Fortune (2025), critical prompt injection vulnerabilities were discovered throughout 2025 in AI coding tools from Cursor, GitHub, and Google's Gemini. Amazon confirmed and mitigated an attempted exploit targeting its Q Developer extension for VS Code. CrowdStrike reported observing threat actors exploiting a code injection vulnerability in Langflow AI, a widely used agent-building tool, to steal credentials.

By June 2025, AI-generated code was adding more than **10,000 new security findings per month** across studied repositories — a 10x increase from December 2024 (SQ Magazine, 2026).

These incidents make clear that the risks of AI in software engineering are no longer hypothetical. They are happening now, at scale, and the engineering profession has not yet fully adapted its practices to match the speed of adoption.
